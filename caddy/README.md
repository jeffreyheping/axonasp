# AxonASP Caddy Module (Experimental)

Welcome to the **AxonASP Caddy Module**! This implementation allows you to run AxonASP directly inside [Caddy](https://caddyserver.com/), a powerful, widely-used, and highly extensible enterprise-grade web server.

> **Note:** This implementation is **still experimental**, so config keys will probably change before the official release to keep it as compatible and easy to use with Caddy server as possible.

## Why Caddy?

Integrating AxonASP with Caddy is highly advantageous. It brings together a modern, robust web server with native support for legacy ASP, VBScript, and JavaScript. The benefits include:
- **Direct Support:** Run your Classic ASP pages natively without complex FastCGI workarounds.
- **Automatic HTTPS:** Caddy handles TLS certificates automatically (e.g., Let's Encrypt), meaning your ASP applications can be secured out-of-the-box.
- **Easy Configuration:** The Caddyfile is notoriously simple and readable, making deployment straightforward.
- **High Performance:** Go-based concurrency combined with AxonASP's optimized Virtual Machine ensures top-tier performance for your legacy web apps.

## How Our Module Works

The AxonASP Caddy module acts as a standard HTTP handler middleware within Caddy. When a request matches an ASP-related route, the module intercepts it:
1. **Request Interception:** The module catches the HTTP request before Caddy attempts to serve it as a static file.
2. **VM Invocation:** It leverages the AxonASP Virtual Machine (`axonvm`) to parse and execute the VBScript/JavaScript embedded in your ASP files.
3. **Response Bridging:** Standard ASP objects like `Request`, `Response`, `Server`, `Session`, and `Application` are emulated seamlessly. Output generated by the VM is written directly to the Caddy HTTP response writer.

## Compiling Caddy with AxonASP

Because Caddy plugins are compiled directly into the server binary, you will need to build a custom Caddy executable containing the AxonASP module.

### Compiling on Windows

You can use the provided PowerShell script (`run_caddy.ps1`) to compile and start the server:

```powershell
# Open PowerShell and navigate to the caddy directory
cd ./caddy

# Run the build and start script
.\run_caddy.ps1
```

If you prefer to compile it manually:
```powershell
go build -o caddy.exe ./cmd/main.go
```

### Compiling on Linux

Compiling on Linux is just as straightforward. Ensure you have Go installed on your system.

```bash
# Navigate to the caddy directory
cd ./caddy

# Build the Caddy binary
go build -o caddy ./cmd/main.go

# Make the executable runnable (if necessary)
chmod +x caddy
```

## Installation & Running the Server

Once you have your compiled executable (`caddy` or `caddy.exe`), you can start the server simply by pointing it to your `Caddyfile`.

```bash
# Linux/macOS
./caddy run --config ./Caddyfile

# Windows
.\caddy.exe run --config .\Caddyfile
```

### Running as a systemd service (Linux)

To run the custom Caddy server in the background as a system service on Linux, you can create a `systemd` unit file. 

1. Move your compiled `caddy` executable and `Caddyfile` to appropriate directories (e.g., `/usr/local/bin/caddy` and `/etc/caddy/Caddyfile`).
2. Create a new service file: `sudo nano /etc/systemd/system/caddy.service`
3. Add the following configuration:

```ini
[Unit]
Description=Caddy with AxonASP
Documentation=https://caddyserver.com/docs/
After=network.target network-online.target
Requires=network-online.target

[Service]
Type=notify
User=caddy
Group=caddy
ExecStart=/usr/local/bin/caddy run --environ --config /etc/caddy/Caddyfile
ExecReload=/usr/local/bin/caddy reload --config /etc/caddy/Caddyfile --force
TimeoutStopSec=5s
LimitNOFILE=1048576
PrivateTmp=true
ProtectSystem=full
AmbientCapabilities=CAP_NET_BIND_SERVICE

[Install]
WantedBy=multi-user.target
```

4. Enable and start the service:
```bash
sudo systemctl daemon-reload
sudo systemctl enable --now caddy
```

## Configuring the Caddyfile

To use AxonASP in your Caddy configuration, use the `axonasp` directive inside your site block.

### Basic Caddyfile Example

```caddyfile
:8080 {
    # 1. Define the document root
    root * ./www
    
    # 2. Set up the route to process ASP pages
    route {
        axonasp {
            site_name my_asp_app
            global_asa_path ./www/
        }
        # 3. Fallback to serving static files (HTML, CSS, JS, etc.)
        file_server
    }
}
```

### Supported Subdirectives

The `axonasp` directive supports the following options:
- `site_name <name>`: An identifier for your application (useful for separating application pools and sessions if running multiple sites).
- `global_asa_path <path>`: The file path to your `global.asa` file, which is used for initializing Application and Session state.
- `config_file <path>`: (Optional) The path to your `axonasp.toml` configuration file.

### Multi-Site Isolation

When hosting multiple applications, the AxonASP Caddy module provides **complete isolation** between sites. By configuring different `site_name` and `global_asa_path` values in separate `route` blocks, each site operates in its own dedicated Application space. This means:
- Each site can have its own distinct `global.asa` file to handle Application and Session events independently.
- Application variables and Session data are fully isolated and do not leak between different sites.

This simple setup will route ASP execution through the AxonASP engine while allowing Caddy's file server to handle your static assets. You can add multiple `route` blocks for different applications or sites, each with its own configuration.
